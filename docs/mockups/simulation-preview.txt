VIEW 0: PklProject not found (shown before simulation if applicable)
================================================================================

When the user runs formae apply/destroy on a .pkl file and no PklProject
file is found by walking up the directory tree:

  ┌─────────────────────────────────────────────────────────────────────────────┐
  │  formae apply                                                               │
  ├─────────────────────────────────────────────────────────────────────────────┤
  │                                                                             │
  │  No PklProject file found in the directory tree for:                        │
  │  ./infra/production/main.pkl                                                │
  │                                                                             │
  │  Formae requires a PklProject file to resolve dependencies. A default       │
  │  PklProject will be created at:                                             │
  │                                                                             │
  │  ./infra/production/PklProject                                              │
  │                                                                             │
  │  Continue? [y/N]                                                            │
  └─────────────────────────────────────────────────────────────────────────────┘

  Notes:
  - Shown as a huh confirmation before simulation starts
  - Shows the path of the pkl file being applied
  - Shows where the PklProject will be created
  - If user confirms: PklProject created, simulation proceeds
  - If user declines: apply aborted
  - With --yes: error out — "No PklProject found. Run formae project init
    or create a PklProject file manually." No silent side effects.
  - Future: this will piggyback on project init to download a best-practice
    template instead of creating a bare PklProject

  │    ~ update     staging-reconcile    auto-reconcile   staging                │
  │    ⊘ detach     old-policy           ttl              staging                │
  │    = keep       shared-policy        ttl              shared                 │
  │                  still referenced by: production, development                │
  │                                                                              │
  │  ▌ Resources                                                                 │
  │    Operation    Label                Type                                    │
  │    - delete     old-cache            AWS::ElastiCache::CacheCluster          │
  │    - delete     legacy-queue         AWS::SQS::Queue                         │
  │                  depends on: old-cache                                       │
  │    ~ update     primary              AWS::RDS::DBInstance                    │
  │    ~ update     web-config           AWS::SSM::Parameter                     │
  │    ↻ replace    web-server           AWS::EC2::Instance                      │
  │    + create     my-bucket            AWS::S3::Bucket                         │
  │    + create     web-1                AWS::EC2::Instance                      │
  │    + create     cdn                  AWS::CloudFront::Distribution           │
  │       ↓ show 10 more (30 remaining)                                          │
  │                                                                              │
  ├──────────────────────────────────────────────────────────────────────────────┤
  │  ↑↓: select  space: expand  →←: column  s: sort  y: confirm  q: abort  ?: help│
  └──────────────────────────────────────────────────────────────────────────────┘

  When a row is expanded (space/enter on "primary"):

    ╭─ ~ primary ──────────────────────────────────────────────────────────────╮
    │ Operation: update                                                        │
    │ Type:      AWS::RDS::DBInstance                                          │
    │ Stack:     production                                                    │
    │                                                                          │
    │ Changes:                                                                 │
    │  ├ set  InstanceClass: "db.t3.medium" → "db.t3.large"                   │
    │  ├ set  MultiAZ: false → true                                           │
    │  ├ add  Tags[backup]: "daily"                                           │
    │  └ remove  Tags[temporary]                                              │
    ╰──────────────────────────────────────────────────────────────────────────╯

  Notes:

  REUSE WITH STATUS DETAIL VIEW:
  This view and the status command detail view (single command subview) share
  the same underlying table component. The differences are:
  - Simulation: operation symbol column (+, ~, -, ↻, ⊘, =) instead of
    state column (✓, ✗, ◐, ○)
  - Simulation: expanded cards show property change diffs (patches)
  - Simulation: confirmation prompt (y: confirm, q: abort) instead of
    live updates
  - Status detail: live state, spinners, duration/age columns
  The shared table component (internal/cli/tui/components/) should be
  parameterized for both use cases — same sort, expand, pagination logic.

  Operation column — combined symbol + word, colored (colorblind-safe):
    + create   (green)  — new resource
    ~ update   (blue)   — property changes
    - delete   (yellow) — destructive but intentional (NOT red — red = failure)
    ↻ replace  (orange) — recreate with different immutable fields
    ⊘ detach   (gray)   — policy being removed from stack
    = keep     (dim)    — policy still referenced, not removed
  Colorblind users read the word; the symbol is a visual shortcut for scanning.
  Yellow for delete distinguishes "planned destruction" from "error/failure"
  (which is red in the status watch view).

  Row coloring:
  - Delete rows: entire row text in yellow/gold (Warning color)
  - All other operations: normal colors (label blue, type gray, op colored)
  - Cursor highlight: full-width background, all text brightened

  Table layout:
  - Same per-group tables as status subview: Targets, Stacks, Policies, Resources
  - Operation column first (symbol + word), then Label, then Type/Stack
  - Policies have Type + Stack columns; Resources have Type column
  - Targets/Stacks have simple Operation + Label layout
  - Elastic columns, truncation with …, same responsive behavior
  - Per-group sorting: →← moves column highlight, s toggles sort
  - Default sort: destructive operations first (delete → replace → update → create)
  - Pagination: show first 10 per group, "show 10 more"

  Expanded cards show property changes (patches):
  - set: path, old value → new value (old in red, new in green)
  - add: path, new value (in green)
  - remove: path (in red)
  - Tree connectors ├/└ for visual hierarchy

  Confirmation:
  - y: confirm and submit the command
  - q: abort
  - With --yes: skip the interactive view, auto-confirm
  - With --simulate: show view but footer says "simulation only" instead of
    confirm prompt

VIEW 3: Large command — handled by pagination (supersedes old auto-collapse)
================================================================================

  Large commands (100+ resources) are handled by the same pagination pattern
  as the status subview: each group shows the first 10 rows, with a
  "↓ show 10 more (N remaining)" row. Default sort puts destructive
  operations first, so deletes are always visible at the top of the group.

  No collapsible tree nodes. No sub-grouping by operation type. Just a flat
  table per group with sort and pagination. The user can sort by Operation
  column to group creates/updates/deletes together if needed.

================================================================================
DATA FIELDS COVERAGE (simulation-specific)
================================================================================

ResourceUpdate fields shown in simulation:
  - Operation ("create"/"update"/"delete"/"replace")
  - ResourceLabel, ResourceType (parenthetical)
  - PatchDocument: rendered as property change tree (set/add/remove)
  - Properties/OldProperties: used for patch rendering context
  - ReferenceLabels: references rendered as labels not ksuids
  - OldStackName: "$unmanaged" triggers "put resource under management"
  - IsCascade/CascadeSource: "(cascade)" label + "depends on" sub-item
  - GroupID: grouped delete+create shown as single "replace" line

TargetUpdate fields shown in simulation:
  - Operation ("create"/"update"/"replace"/"delete")
  - TargetLabel
  - Discoverable (for updates: "to discoverable" / "to not discoverable")
  - Replace shows reason (immutable field change)

StackUpdate fields shown in simulation:
  - Operation, StackLabel, Description (parenthetical)

PolicyUpdate fields shown in simulation:
  - Operation ("create"/"update"/"delete"/"detach"/"skip"→"keep")
  - PolicyLabel, PolicyType, StackLabel (parenthetical)
  - PolicyConfig/OldPolicyConfig: config changes as tree (for updates)
  - ReferencingStacks: for "keep" operations

Fields not relevant for simulation:
  - State, Duration, CurrentAttempt, MaxAttempts, StatusMessage,
    ErrorMessage: all pending, not yet executed
  - ResourceID: internal
  - StartTs/ModifiedTs: not started yet
